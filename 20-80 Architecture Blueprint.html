Strategic thesis from the meeting

AI is making web development itself cheap and fast — that part of the business will commoditise. What clients actually pay 20-80 for is strategic, industry-specific judgement: healthcare marketing, AHPRA compliance, patient psychology, conversion, Ads/SEO strategy, growth advice. So the plan is to automate the repetitive delivery layer, keep the strategic layer human, serve more clients with the same team — and eventually productise the whole system as a SaaS + training offer for other agencies (a high-ticket program, ~$30k with platform access and workshops was floated).

Bottlenecks → automations

SLIPPING

Client emails to support@ fall through the cracks; tasks are created manually; clients wait too long to hear their job is done.

W2 · INBOX AI

AI monitors the inbox, creates and assigns ActiveCollab tasks, tracks completion, and drafts the "it's done" reply automatically.

Client-facing product deployed on each practice's phone line: answers as the practice in EN/中文, captures booking requests, feeds the front-desk follow-up list. Rated the #1 quick win — and the first sellable product.

MANUAL

Google Ads and SEO performance reviewed by hand; BrightLocal checked client-by-client; drops surface late at support meetings.

W3 + W4 · MONITORS

The existing Ads benchmark dashboard gains AI flagging and suggested optimisations; BrightLocal feeds rank-drop alerts proactively.

LOST

Fathom records meetings but action items are moved to ActiveCollab by hand — or not at all.

W5 · MEETING AI

Transcript → Claude → tasks in ActiveCollab + client summary email + Slack notification.

SEMI-MANUAL

Onboarding info still processed by hand; SEO metadata set up via spreadsheets and a one-off Claude prompt.

Onboarding form generates a build-ready PRD; the SEO-metadata prompt becomes a reusable dashboard function.

A simplified AI-powered CMS: client-safe content-only editing, AI-assisted updates, one-click publishing.

The SaaS / training play

Build for 20-80 first → prove with real case studies → refine → wrap as SaaS → sell with training, coaching and support. The software is not the product; the knowledge base and proven workflows are. Pricing floated: a high-ticket program around $30k including platform access and workshops. Architecturally this costs nothing extra now — it just requires the discipline of keeping 20-80's rules (philosophy, SEO, AHPRA, design) in the versioned knowledge base rather than hard-coded into workflows, so another agency's rules can be swapped in later.

09

Decisions needed before build

Telephony platform for the AI receptionist.

Voice AI provider and number routing (e.g. forward-on-no-answer vs full virtual reception), and whether Chinese-language voice quality meets the bar — worth a bake-off before committing.

Google Ads execution path.

Claude MCP for Google Ads vs native Ads API + scripts vs Markifact (impressive, but credit costs flagged in the meeting) vs local browser automation. Recommendation: start read-only via the Ads API for flagging, choose the write path once the approval queue exists.

BrightLocal access level.

The API requires a suitable plan tier. If unavailable, Claude can parse scheduled CSV/email reports as a bridge — the architecture supports both, but the API is far more reliable.

Auto-send thresholds for client comms.

Which AI-drafted messages may send without human approval, and when? Suggested: everything human-approved for the first 90 days, then auto-send low-risk categories (task-completed confirmations) with spot-checking.

Benchmark definitions.

What "below benchmark" means per practice type (CPL? leads/month? the 10:1 ROI line?) — one workshop to set initial thresholds, tuned once data flows.

Chinese variant & reviewer.

Simplified, Traditional, or both — and who approves ZH content in the loop before the glossary matures.

CMS scope & Emergent exit.

Greenfield CMS vs headless layer over existing sites, and the migration path for the 100+ live sites. This decision sizes Phase 5 more than any other.
