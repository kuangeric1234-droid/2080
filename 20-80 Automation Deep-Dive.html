W1 · CLIENT-FACING PRODUCT — DEPLOYED PER PRACTICE

AI Receptionist for clinics — the practices' phones, not ours

Deployed on each client practice's phone line: answers calls the front desk misses (busy, lunch, after hours, weekends), speaks English or 普通话 based on the caller, handles patient FAQs from the practice profile, captures appointment requests, and hands the front desk a clean follow-up list. 20-80's dashboard gets a fleet view across every deployed practice — uptime, calls saved, bookings captured — which is also the proof for selling it as Product 2.

TRIGGER: practice line unanswered in N rings · after-hours / weekend schedule · lunch overflow SKILLS: clinic-call-agent (per-practice config) · call-summary WRITES: practice portal follow-up list · SMS to caller & front desk · practice PMS/booking (phase 2: Praktika, Core Practice, HealthEngine) HARD RULE: never gives clinical advice — ever FLEET: agency dashboard monitors every deployment

Edge cases & failure handling

Genuine medical emergency
Triage script only, immediately — approved wording per practice, P1 SMS to the nominated contact, full recording retained. The agent never assesses severity itself beyond keyword triggers; it errs toward the urgent path.
Health information privacy
Minimum-necessary capture (name, contact, preference — not clinical detail); recordings on a practice-set retention clock; transcripts auto-redact beyond what the follow-up needs; data stays per-practice, never pooled.
Caller hangs up mid-capture
Partial summary still filed with caller-ID; optional "sorry we missed you — book here" SMS if the practice enables it.
Cantonese caller
v1 voices: EN + Mandarin. Cantonese detected → bilingual capture prompt + follow-up item tagged for the practice's Cantonese speaker.
Voice platform outage
Fleet health-check every 5 min; on failure the line falls back to the practice's ordinary voicemail; voicemails transcribed on recovery into the same follow-up list. 20-80 gets the outage alert, not the practice's patients.
Prank / marketing robocalls
Classified spam → polite close, logged, excluded from stats so the practice's "calls answered" number stays honest.

Edge cases & failure handling

Client emails a team member directly
House rule: forward/BCC to support@. Nightly sweep also checks connected personal inboxes for client-domain mail with no matching task and nudges the owner in Slack.
Reply-chain noise ("thanks!")
Thread-aware: replies on a tracked thread with no new request are filed to the timeline, never re-triaged into duplicate tasks.
Ambiguous client (referrer, new prospect?)
No domain match → prospect flow: lead record + Wally notified, not a support task.
Request partially completable
Each split request has its own task and its own completion email — one blocked item never silently delays the others (see the Dr Wong bio chase).
Out-of-office autoreply loops
Auto-reply headers detected → suppressed from pipeline.
Gmail push outage
Fallback poll every 5 min; watch-channel renewal monitored; gap detection compares message counts and backfills.

Edge cases & failure handling

OAuth token expiry / account unlinked
Nightly connection audit; failure → red banner on account tile + task, and the account is marked "data stale" so no recommendations are generated from old data.
API quota exhaustion
Pulls are batched and staggered; on quota error, exponential backoff and the run resumes — a partial morning run shows "7/15 accounts refreshed" honestly, never silently.
Conversion lag
Dental bookings convert days later; 7-day windows use lag-adjusted conversions (Ads API conversion_lag_bucket) to avoid flagging fresh campaigns as failing.
Client requests a change directly (via W2)
ads_request tasks route to the specialist, and any resulting change goes through the same diff + audit path, so the audit log is complete regardless of who initiated.
Two changes overlap
Change-watch attributes outcomes to change *sets*, not single edits; overlapping windows are merged into one evaluation to avoid false rollback proposals.
Markifact / MCP evaluation
The recommend→approve→implement contract is vendor-neutral: the implement step can be Ads API, Google Ads scripts, or an MCP server — swappable without touching the loop.
