## 1.4 · Entity matcher + timeline — 2026-07-07

**Built:** `server/src/matcher.ts` + migration `0002_match_queue.sql`. Matcher implements SPEC-SPINE §3 exactly, deterministic first: ① exact source-ref maps (phone DID / Ads / BrightLocal / GBP / portal session / AC project) → 1.0 · ② contact email exact ∪ learned email mappings → 0.98, ties always queue · ③ thread continuity → 0.97 · ④ domain (entity_maps ∪ contact-email domains) → 0.90, shared domain never autos · ⑤ fuzzy signature-name/phone → ≤0.7, always queues · nothing → prospect. `ingestEvent` routes inbound events: auto-attach to `timeline_events` (learning the thread mapping so stripped replies match on continuity), hold in `match_queue`, or park as prospect for the sales flow. `resolveMatch` turns a human's queue choice into an `entity_maps` row — the matcher learns — then attaches the held event and audits. New `match_queue` table because `timeline_events.client_id` is deliberately NOT NULL: unmatched events never touch the timeline.

**Evidence:** golden set `server/test/golden/matcher.golden.json` — 30 inbound events across all five tiers routed **30/30** (DoD floor ≥29/30 = 95%) · every queue decision confidence <0.8, every auto ≥0.8 · shared-domain case queues with both candidates, never guesses · queue resolution learns: bentrowse@gmail.com queued (fuzzy 0.6) → resolved to Trowse → same sender autos at 0.98 next time · resolved items can't resolve twice · prospects held in queue state `prospect`, off the timeline. Full server suite 30/30, app 8/8, typechecks clean.

**Files:** `server/src/matcher.ts`, `server/migrations/0002_match_queue.sql`, `server/test/{matcher.test.ts, golden/matcher.golden.json}` (new) · `seed.ts` (contact phone numbers added for the fuzzy tier).

**Decisions:** (1) `match_queue` is a new table the spec's "human queue" implies but §2 doesn't name — refs + held event + candidates, states open/resolved/prospect/dismissed. (2) Shared domains are detected via contact-email domains (entity_maps has a uniqueness constraint per kind+key, so a shared domain can't be two mapping rows — the management-group scenario lives in contacts). (3) On resolve, the most specific ref is learned (email > phone), plus the thread. (4) Match-queue UI lands with Inbox (1.5), which is this queue's production surface.
